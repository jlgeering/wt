# ADR: `wt switch` Command Design

- Status: Accepted
- Date: 2026-03-15
- Decision Makers: wt maintainers

## Context

`wt` supports interactive worktree switching via bare `wt` (shell wrapper picker). Scripts,
tools, and agents that already know the destination branch have no direct non-interactive path —
they must simulate input or parse `wt list` output themselves.

We need a named switch command for the programmatic use case.

## Decisions

### 1. `wt switch <branch>` prints path to stdout in both human and machine modes

Unlike `wt new` (which suppresses the path in human mode to avoid noisy interactive output),
`wt switch` has no side effects to narrate. The path is the only meaningful output. Printing
it unconditionally enables `cd $(wt switch foo)` without shell integration, which is the
primary use case.

The shell wrapper always calls `wt __switch` (machine variant) to be explicit, but the
output contract of `wt switch` (public) is identical.

### 2. `wt switch <branch>` errors if no worktree exists — no creation fallback

Creating a worktree has config implications: `.wt.toml` copy rules, base ref selection, and
setup hooks. Mixing creation into a lookup command would either silently ignore those concerns
or require duplicating `wt new`'s complexity.

Scripts that want create-or-switch can chain explicitly: `wt switch foo || wt new foo`.

"No worktree exists" means no entry from `git worktree list --porcelain` reports that branch name.
Detached-HEAD worktrees are therefore ignored by `wt switch`, even if their `HEAD` commit happens
to match the requested branch tip.

### 3. `switch` is a public command listed in `wt --help`

The command is useful without shell integration (scripts, CI, agents). Making it discoverable
via help is consistent with the tool's philosophy of explicit, inspectable interfaces.

The internal `wt __switch` variant exists so shell wrappers have a stable machine-contract
boundary, matching the `__new` / `__list` pattern.

## Consequences

- Scripts and agents get a direct, unambiguous way to resolve branch → worktree path.
- `wt switch` stdout behavior differs from `wt new` stdout behavior — documented here to
  prevent future confusion.
- Detached-HEAD worktrees remain visible in `wt list`, but they are not valid `wt switch` targets.
- No create-or-switch shorthand; callers compose explicitly.
